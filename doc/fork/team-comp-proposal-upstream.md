# 提案：逐光捡金任务的配队能力分级方案（供维护者讨论）

> 这份文档是为和上游维护者讨论准备的，自包含、可直接贴进 issue/discussion。
> 背景：我已提交逐光捡金三任务（PF/MoC/AS）的 PR（固定两个预设编队序号 + 按策略选关 + 自动战斗），
> 即 #298 中方向 1 的实现。本提案讨论的是**下一步：从"固定一对编队"到"按关卡换编队"**，
> 按维护成本分了四档，每档独立成立、独立可停，默认行为全部不变（opt-in）。

---

## 给维护者的一句话版本

所有方案都遵守同一条红线：**不引入任何"会卡住版本更新的人工策略数据"**。
方案 1/2 完全零角色级数据（标签由用户贴在自己的候选队上）；方案 3 引入的角色体系标签
JSON 是**可选新鲜度**——新角色没人补标签时自动降级为元素匹配照常工作，补标签是社区
一人一行的可选贡献（MAA recruitment.json 模式），不是维护者义务。角色属性映射用项目里
已有的 `CharacterList` 关键词（`keyword_extract` 每版本例行再生，零边际成本）；
识别全部基于屏内信息（弱点图标/OCR），不依赖外部 API、不需要米游社登录态、不联网拉数据。
实现、双账号实测、离线回归测试都由我（贡献者）完成，分独立 PR 提交，您每档审完可以随时喊停。

## 方案 1：候选编队对轮换（零识别、零数据）

把配置从"一对预设序号"扩成"多对预设序号的有序列表"（沿用 filter 字符串习惯，如
`1-2 > 3-4 > 3-2`，用户保证每对内部角色不重叠）。运行逻辑：

- 按序使用第一对；某关失败重试时，先**交换节点 1/2 的队伍顺序**再打一次
  （两半场敌人属性不同，换序的实战收益很高，March7thAssistant 已采用此做法）；
- 再失败换下一对候选；候选与预算用尽才走现有 RetryExceeded（明天再试/本周放弃）。

**这同时就是"凹分模式"**（#298 下有用户求过凹分循环）：不同配队适应不同环境但先验
无法知道，玩家把可能的配队全列进候选，程序按序穷举**直到达成目标星数**，真值由实战
裁决。与现有语义零冲突——`push`/`sweep` 本来就是"未满星才选中"（隐含目标满星），
`first_clear` 目标通关；凹分 = 现有模式 + 候选穷举 + 预算放大，**不新增模式名**。
两个细节：穷举是两层嵌套（组合顺序 × 每组合可重试 k 次，容纳暴击/AI 走位的 RNG 波动，
社区"凹"的本义就包含同队重打）；游戏保留历史最佳记录，凹分只赚不亏。单关预算直接
复用 MaxRetry（语义自然从"重试同一对几次"泛化为"该关总尝试预算"），调度衔接不变。

**Diff 范围**：argument.yaml 一个字符串参数 + 选编队点位的轮换逻辑，无新资产。
**价值**：低练度账号"换一套队再试"从无意义变为可能；卡关不再死磕同一对；凹分党把
MaxRetry 调大即得穷举凹分。

## 方案 2：标签匹配选队（体系 + 属性，用户标注）

设计前提：**命途不能定义队伍功能**（后期角色功能与命途普遍脱钩，同谐里有击破开关位、
丰饶里有击破奶），玩家社区实际使用的分类是**体系**：击破/超击破/DOT/追击/记忆/群攻等。
体系词表本身放数据文件，加新体系不改代码。

在方案 1 的候选对上，用户给每队贴体系和/或主C属性标签（形态可议，示意：
`1-2:超击破/火 > 3-4:记忆/冰`）。进入关卡准备屏后（现有流程本来就经过这里）按四层打分：

1. **模式先验**：末日幻影机制天然偏击破系、虚构叙事天然偏群攻/记忆——游戏机制层面的
   稳定事实，免费且永不过期；
2. **弱点命中**：模板匹配节点敌方弱点图标（7 种属性，常年稳定的一次性资产），主C属性命中加分；
3. **增益亲和**：OCR 虚构叙事增益/末日幻影公理/忘庭乱流文本，按"体系×关键词"亲和表加分
   （如文本含"击破"→击破系队加分），同一张表顺带解决"增益/公理选哪个"（当前写死选第一个）；
4. **用户顺序**：候选顺序即用户的练度/偏好声明，做先验分。

选出"两节点总分最大且不重叠"的分配；失败重试时该 (队,节点) 组合降权，自然轮换。

要点：**打分只决定穷举顺序，不做一锤定音的选择**。底下永远是方案 1 的凹分循环兜底，
打分失误的代价是多花时间而不是打错收场——这让规则表不必追求精确，维护压力进一步降低。

**Diff 范围**：7 个图标资产 + 打分函数 + 标签解析 + 一张亲和小表（数据文件）。
**维护面**：亲和表/词表常年不动（体系×关键词的映射不随版本变）；**零角色级数据**。
**先例**：StarRailOneDragon 的逐光捡金用同构方案（职能/属性标签 + 弱点匹配 DFS 打分）跑在生产环境。

## 方案 3：预设编队自动识别（用户零标注）

去掉方案 2 的手工标签：程序自己认出每个游戏内预设里是哪 4 个人——

- **属性自动**：查 `CharacterList`（已有数据，零新增维护）；
- **体系自动**：新增一份角色体系标签 JSON，每角色一行
  （`role: 主C/副C/增幅/生存` + `archetypes: 体系列表`，辅助角色的 archetypes 表示"兼容哪些体系"，
  `*` 为泛用）。**队伍体系 = 主C的体系**，成员兼容度做质量加成——主C定性简单且符合玩家直觉。

识别两条实现路线供选择：

- **3a 头像模板**：打开预设面板一屏识别所有候选（快）。需要角色小头像资产，
  建议做成生成器：从数据源批量裁切生成，并入 `keyword_extract`/`button_extract`
  同级的版本例行流程——新角色上线跑一次生成器即覆盖，无人工劳动。
- **3b 套用后 OCR**：逐个套用预设后用 `OcrCharacterName`（`switch.py` 现成类）读上阵区
  角色名（慢，每个候选多 2~3 秒，但**零新增资产**）。

**体系标签 JSON 的治理（本方案唯一的新数据面）**：

- 放仓库内，社区 PR 维护（一个新角色一行，MAA 的 recruitment.json 模式已验证可行）；
- 用户本地有同构 override 文件，可覆盖默认值（练度因人而异，"我的XX是辅助练法"这类个性化）；
- **缺失自动降级**：新角色没人补标签时，按"属性匹配 + 命途猜生存位"兜底照常工作
  （命途虽不能定义体系，但丰饶/存护≈生存位的兜底猜测准确率仍高，仅在缺数据时启用）——
  **数据新鲜度影响效果上限，不影响可用性，永远不卡版本更新**。

**价值**：用户只需在游戏里建好预设（这本来就是玩家自己会做的事），SRC 侧不再要求
用配置重复描述队伍；同时为后续可能的功能（如忘庭乱流适配）留下感知基础。

## 失败兜底（各方案共用）

识别失败/标签缺失/候选耗尽时，一律降级到上一档行为，最终落回现状（固定第一对）。
自动选择永远只在"用户给出的候选"里挑，不会自作主张组新队——配队的责任边界仍在用户。

## 为什么不做模拟/抄作业/外部 API（与 #298 立场一致）

- **战斗模拟**：调研结论是 HSR 没有可持续维护的开源战斗模拟器，唯一较完整的
  [Star-Rail-Battle-Simulator](https://github.com/Darkglade1/Star-Rail-Battle-Simulator) 已停更 16 个月，
  此类项目普遍死于"每 6 周人肉移植新角色机制"。自建等于无限维护承诺，不符合本项目体量。
- **抄作业/使用率统计**：养成因人而异（#298 原话），高频队伍 ≠ 用户练度能复刻；
  且依赖第三方数据源（个人项目，寿命不可控）。
- **固定队伍模板库**（社区维护"超击破=流萤+同谐主+阮梅+(加拉赫|灵砂)"这类带替补组的成队模板）：
  在"从用户预设里选"的形态下没有必要——用户的预设就是用户的固定队伍，工具要做的是给它们
  自动定性和匹配（方案 2/3 已覆盖）；模板库的价值在"替不会配队的人从 box 组队"，那是另一个
  量级的功能（编队界面自动组任意队），不在本提案范围。
- **米游社 API**：需要用户 cookie，登录态/DS 算法是额外维护面，且涉及账号风险敏感性。
  这类增强适合留给下游/个人 fork，不进上游。
