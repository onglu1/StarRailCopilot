# 深渊自动配队方案调研

> 2026-06-12。背景：深渊三任务（PF/MoC/AS）目前只支持两个固定预设编队序号，难度各异、复杂度高，
> 低练度账号很难满星。目标是演进到"灵活配队 + buff 选择"，终极目标是零配置全自动。
> 本文是方案调研，未做实现决策。结论先行：见文末「推荐路线」。

## 0. 问题定义

把"自动配队"拆成四个独立子问题，方案的差异本质是各子问题由谁回答：

| 子问题 | 说明 |
|---|---|
| ① 我有什么 | 玩家可用角色及练度（候选池） |
| ② 关卡要什么 | 当期敌方弱点属性、机制、增益（虚构叙事增益/末日幻影公理/忘庭乱流） |
| ③ 什么队好 | 队伍内部协同（谁和谁搭）+ 队伍对关卡的适配 |
| ④ 怎么分配 | 节点 1/2（星启 3 节点）角色不重叠，最大化整体把握 |

④ 是纯算法问题且已证明简单（每节点枚举 Top-K 候选队做 bitmask 不相交扫描，毫秒级）。
难点全部在 ③，①② 是数据获取问题。

### 1.3 其他游戏的模式归纳

- **MAA(明日方舟)**：作业站众包关卡作业；作业协议用 `groups`——一个位置写**候选干员组**，程序按练度自动挑一个。肉鸽招募优先级是仓库内社区维护的权重 JSON。→ 可借鉴：**写候选组而非锁死角色；优先级表做成用户可改的本地数据，开发者只给默认值**。
- **BetterGI(原神)**：不自动配队；OCR 识别当前队伍，套社区共享的战斗策略脚本，脚本里不在队的角色自动忽略 → 一份通用配置自适应任意 box。
- **Fribbels HSR Optimizer**：练度评分不是回合模拟，是**固定轮转公式+词条最优基准的百分比对比**；角色模型开发者维护、测试服阶段即跟进；纯浏览器无 headless API。
- **TFT 阵容推荐(Blitz/Mobalytics)**：终盘阵容聚类出 meta 原型 → 按玩家当前棋子与原型的**重合度**推荐转型。纯统计自下而上，零机制知识。
- **FGO Chaldea Laplace**：玩家实际通关后上传"配队+操作"，他人按关卡浏览，客户端按自己 box **本地重算验证**可行性；不做自动求解。
- **WoW SimulationCraft**：全模拟路线只有大型志愿者社区养得起（每职业专人维护，弃维护的模块直接烂掉）。

### 1.4 战斗模拟可行性 → 判死刑

- HSR **没有 gcsim 等价物**。唯一较完整的开源回合模拟器 [Star-Rail-Battle-Simulator](https://github.com/Darkglade1/Star-Rail-Battle-Simulator)（Java，22 角色）2025-02 停更；玩具级项目普遍 6-12 个月弃坑。死因一致：**每 6 周人肉移植新角色机制**。
- 敌人数值/排期拿得到（datamine），**拿不到的是机制语义**：敌人 AI、增益文本→可执行逻辑，全靠人肉编码。
- 闭源网页 [starguide.gg](https://starguide.gg/) 已实现"导入 box → 启发式协同评分+练度修正 → 三模式选不重叠两队"，证明**零配置不需要真模拟，启发式+统计就够**。
- 结论：自建模拟器 = 给自己签每 6 周的劳役合同，**不做**。打分函数只能走 规则/标签（A）、统计（B）、启发式（C）。

## 2. 可行思路（按配置成本递减、实现成本递增排序）

### 思路 A：候选队池 + 标签/弱点规则匹配（一条龙模式）⭐ 推荐主线

**原理**：把配置从"2 个预设序号"升级为"N 个带标签的候选队"。程序在选关/准备屏识别
②（弱点图标模板匹配 + buff 文本 OCR 关键词），按规则给每个候选队×节点打分
（主C属性命中弱点 ≫ 标签与增益关键词匹配（如增益含"击破"→击破队加分）≫ 生存位齐全），
然后解不重叠分配。失败重试时换下一对组合而不是死磕同一对（现有 MaxRetry 记账直接复用）。

两档实现：

- **A1 零视觉版**：标签写在配置里（每个预设序号附 `主属性/流派标签`），或者直接用**游戏内预设编队的名字当标签**（用户把预设命名为"火击破""DOT"，程序 OCR 预设名）——不加任何新识别，半天可做。
- **A2 自动识别版**：模板匹配预设行内 4 个小头像（StarRailRes 头像库 vendor 到本地），查静态表得到每人属性/命途 → **属性覆盖自动算出**，用户只剩"流派标签"可填可不填。每个新角色上线只需补一行静态数据（可从 StarRailRes 自动生成，几乎零维护）。

**①来源**：游戏内预设编队（用户建好即声明"我有且练了"）。**②来源**：屏内识别，零外部依赖。
**每期维护成本：零**。**依赖：无**（StarRailRes 静态文件 vendor 一次）。
**风险**：识别坐标标定工作量（弱点图标、预设头像）；标签规则表是手写的（但很小，流派×增益关键词矩阵）。
**先例**：一条龙生产环境验证过同构方案；SRC 维护者认可的方向 1 的自然延伸。

### 思路 B：统计驱动（TFT/Prydwen 模式）

**原理**：拉当期 [MocStats](https://github.com/LvlUrArti/MocStats) 使用率数据 → 高频四人组/成对队伍按
层数排序 → 与玩家候选池求交（B1: 池=预设编队，识别同 A2；B2: 池=米游社 API 全量 box）→
推荐重合度最高的可组建队伍。若数据含成对记录（两队同场清关），不重叠问题被真实玩家数据天然解决。

**变体 B0（被低估，强烈建议纳入）**：只拉**玩家自己的历史深渊战报**（genshin.py `challenge*`）——
上一期你自己用什么队过的哪一层、几星，全有。当期弱点相近时直接复用自己验证过的队。
零外部 meta 依赖、完美匹配练度、绝对"会用"。

**每期维护成本：零**（数据自动更新）。**依赖**：网络拉取；B0/B2 需用户填米游社 cookie（只读接口，封号风险极低）。
**风险**：第三方数据源寿命（MocStats 个人项目）；国际服样本对国服 meta 的偏差（小）；
高频四人组 ≠ 你的练度能复刻（需叠加 A 的规则兜底）。

### 思路 C：练度感知启发式（starguide 模式，终极方案的务实形态）

**原理**：全量 box+练度（米游社 API 或 reliquary-archiver）→ 粗粒度练度分
（等级/星魂/光锥/遗器主词条，不做轮转模拟）→ 角色职能/搭档静态表（每角色一行：定位、
最佳搭档、流派）→ 枚举组队 → 评分 = 流派完整度 × 弱点命中 × 练度 × meta 先验(来自B) →
不重叠求解。**完全脱离预设编队**，自动在编队界面组任意队（滚动列表+头像模板匹配选人，
March7th 已验证此交互可行，不需要中文输入）。

**每期维护成本：零；每新角色成本：静态表加一行**（角色定位/搭档，约几分钟，可参考社区 tier 表）。
**风险**：评分函数手感需要调；编队界面自动组队是新的大块 UI 自动化；
"自动组的队用户没练过手法"在 AUTO 战斗下问题不大但满星率未知。
**定位**：A+B 落地且好用之后的远期目标，不建议首发。

### 思路 D：战斗模拟驱动 → 不做

见 1.4。唯一复活条件：社区出现可持续维护的开源 HSR 战斗模拟器（目前没有迹象）。

## 3. 与思路无关的三个快赢（建议无论选哪条都先做）

1. **失败换队反馈环**：当前 MaxRetry 重试打同一对编队。改成：重试时先交换节点 1/2 队伍顺序
   （March7th 做法，半场敌人属性不同，换序经常就过了），再败换下一对候选。改动只在
   `abyss_run_challenges` 的重试分支，一两小时的活。
2. **buff/公理关键词选择**：当前写死选第一个。OCR 增益卡片文本 → 与队伍标签做关键词匹配
   （"击破特攻"→击破队标签，"持续伤害"→DOT 标签），选不中再回退第一个。规则表 10 行以内。
3. **星数感知的策略升级**：现在 first_clear 只看 0 星。配合 A 的多候选队，"2 星关换一套队再试"
   成为可能（之前没得换，重试无意义）。

## 4. 推荐路线

```
第一步（快赢包）   失败换序重试 + buff 关键词匹配          零新依赖，纯 fork 层
第二步（主线）     A1 标签配置 → A2 预设头像自动识别        每期零维护，无 cookie 无网络
第三步（增强）     B0 自己的历史战报复用（cookie 可选项）    个性化最准的数据白嫖
第四步（增强）     B1 MocStats meta 匹配（联网可选项）      "抄作业"但按你的池子抄
远期（可选）       C 全量 box 启发式 + 自动组任意队         starguide 的本地开源版
不做              D 战斗模拟
```

**架构约束**（呼应上游 #298 的表态）：配队智能全部放 **fork 层新文件**（如 `tasks/abyss_fork/`
或 `tasks/abyss/composer.py` 独立模块，按 PR 边界定），通过子类/注入挂到现有
`abyss_run_challenges` 的"选编队"点位上；`pr/abyss` 分支保持纯净（固定预设序号），
上游若收编也不影响 fork 层叠加。所有外部数据（米游社/MocStats/datamine）做成
**可选开关 + 本地缓存 + 失联降级**（降级到 A 的纯屏内识别路径）。

## 6. 设计深化：标签体系（2026-06-12，用户反馈后定型）

用户反馈两点：**命途是伪信号**（后期角色功能与命途脱钩，不得作为体系依据；唯一残值是
"丰饶/存护≈生存位"的缺数据兜底猜测）；**体系才是玩家真实使用的分类**（击破/超击破/DOT/
追击/记忆/群攻……），通常由玩家社区维护。属性（弱点元素）仍是硬信号，与命途拆开对待。

**标签 vs 固定队伍的裁决**：在"从预设里选"的产品形态下，固定队伍已经存在——用户的游戏内
预设就是其固定队伍（建预设=同时声明"成队+练了+会用"）。工具要做的是给预设**自动定性**
（什么体系）+ **加权匹配**关卡，而不是发明队伍。社区固定队模板库（带替补组的成队模板，
MAA groups 模式）的价值只在"替不会配队的人从 box 组任意队"（C 路线）兑现——延后，
不进上游提案。结论：**标签是引擎，预设是固定队，模板库是 C 的数据层**。

**数据模型**（仓库 JSON + 用户本地 override 同构文件）：

- `vocab`：体系词表本身是数据，加体系不改代码。v1 建议：击破/超击破/DOT/追击/记忆/群攻/站场。
- `characters`：每角色一行 `role(主C/副C/增幅/生存) + archetypes`（主C="我是什么"，
  辅助="我兼容什么"，`*`=泛用）。**队伍体系 = 主C的体系**，成员兼容数做质量加成——
  比四人标签投票干净，符合玩家直觉。
- `affinity`：体系×关卡特征权重（buff 关键词、模式先验），常年不动。

**打分管线**：模式先验（AS偏击破/PF偏群攻记忆，游戏机制级稳定）→ 主C属性∈节点弱点
（CharacterList 零维护）→ 体系×增益/公理/乱流 OCR 关键词亲和（同一张表顺带解决
"buff 选哪个"）→ 队伍完整度（有主C有生存）→ 用户候选顺序先验 → 不重叠分配；
失败重试对 (队,节点) 组合降权轮换，衔接现有 MaxRetry 记账。

**不卡版本原则**：新角色缺标签 → 自动降级为属性匹配+命途猜生存位，可用性不受影响；
补标签是社区一人一行的可选贡献（MAA recruitment.json 先例），不是维护义务。
上游提案（[team-comp-proposal-upstream.md](team-comp-proposal-upstream.md)）已按此重写：
方案 2=用户给候选对贴体系/属性标签（零角色级数据），方案 3=识别+角色标签 JSON（带降级），
模板库列入"明确不做"。

**凹分模式（用户提出，定型）**：不同配队适应不同环境但**先验不可知**——玩家列出全部
可能配队，程序按打分排序穷举，直到达成目标星数。架构意义：把打分从"选择器"降级为
"排序器"，选错的代价从打错变成多花时间，真值由实战裁决——配队问题最大死穴（先验
不可知）被搜索闭环治掉，规则表不必精确。与现有语义零冲突：push/sweep 本就"未满星才
选中"（隐含目标满星）、first_clear 目标通关，凹分=现有模式+候选穷举+预算放大，
